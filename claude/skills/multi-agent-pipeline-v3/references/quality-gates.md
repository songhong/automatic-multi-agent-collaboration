
# Quality Gates Protocol

## Two-Layer Quality Model

The pipeline uses two separate quality gates:

1. `completion_quality_gate` runs for every task. It verifies correctness, requirement coverage, completeness, runnable output, maintainability, and whether the task package was sufficient.
2. `premium_review_gate` runs only for key batches and final delivery. It reviews professional polish, consistency, user experience, documentation readiness, visual/aesthetic quality, performance, accessibility, and cross-deliverable coherence.

Do not run premium review on every small task by default. It is a quality amplifier, not the normal bug check.

## Planner Responsibilities

For every task in the queue, planner must write:

```json
{
  "quality_gate": "completion_quality_gate",
  "premium_review_required": false,
  "premium_review_reason": "N/A",
  "quality_acceptance_criteria": [
    "The output satisfies the user-visible goal, not only the minimal implementation step.",
    "All relevant source anchors and material paths have been covered.",
    "The result is usable by the next agent or by the user without guessing."
  ],
  "parallel_group_id": "parallel-group-001",
  "dependency_task_ids": [],
  "conflict_risk": "low | medium | high",
  "shared_output_paths": []
}
```

Set `premium_review_required: true` when a task or batch is a homepage, final report, README, public-facing document, core interaction, visual design surface, user explicitly asks for high quality, or the batch is the final delivery candidate.

## Tester Responsibilities

Every tester report must separate issues into:

```markdown
CORRECTNESS_ISSUES:
- ISSUE_ID:
  SEVERITY: blocking | major | minor
  DESCRIPTION:
  SUGGESTED_FIX_DIRECTION:

QUALITY_ISSUES:
- ISSUE_ID:
  SEVERITY: blocking | major | minor
  DESCRIPTION:
  SUGGESTED_FIX_DIRECTION:
```

A tester may PASS only when both correctness and quality issues have no blocking or major items for that tester's scope. Minor polish may be recorded as non-blocking only when it does not affect the user's stated success criteria.

## Premium Review Routing

For key batches, coordinator dispatches only the premium-relevant testers named by planner. Typical mapping:

- UI/product surface: `tester-visual-aesthetic`, `tester-accessibility`, `tester-runtime-effect`, optionally `tester-performance`.
- Documentation/report/deck: `tester-visual-aesthetic`, `tester-data-integrity`, `tester-code-quality` when generated by code.
- Backend/data/API: `tester-runtime-effect`, `tester-security`, `tester-performance`, `tester-data-integrity`.
- Final delivery: `release-packager` performs final audit and may route failures to the responsible developer, integrator, tester, or planner by path.

Premium review failures enter the same repair loop: the original developer repairs, the original tester retests, and three failed repair attempts stop the batch for human review.
